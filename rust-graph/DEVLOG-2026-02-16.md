# Dev Log: symbol_extractor.py — 2026-02-16

## Goal

Extract full symbol hierarchy from Rust source files via rust-analyzer LSP, producing a JSON tree per file with structs, enums, functions, methods, fields, their line ranges, and optionally full type signatures from hover.

## What was created

### `rust-graph/symbol_extractor.py` (new)

Core module with the following API:

- **`_convert_symbol(sym, hover_fn=None, relpath=None)`** — recursively converts LSP `DocumentSymbol` into our node format. Fetches hover signatures only for Function (kind 12) and Method (kind 6).
- **`_build_file_tree(relpath, symbols, total_lines, hover_fn=None)`** — wraps converted symbols in a File root node.
- **`extract_file_symbols(client, relpath, signatures=True)`** — extracts the full symbol tree for one file.
- **`extract_symbols_iter(client, file_relpaths, signatures=True)`** — generator that yields one file tree at a time (streaming).
- **`_find_workspace_rs_files(project_root)`** — uses `cargo metadata --no-deps` to discover only workspace crate source directories, then walks those (skipping `target/` and hidden dirs). Much more targeted than a naive `os.walk`.
- **`HOVER_KINDS = {6, 12}`** — public constant for which symbol kinds trigger hover calls.

CLI usage:
```
python3 symbol_extractor.py <file1> [file2 ...]
python3 symbol_extractor.py --all-rs                # workspace .rs files only
python3 symbol_extractor.py --all-rs --no-signatures # skip hover (much faster)
```

Environment variables (from `lsp_client.py`):
- `RA_PROJECT_ROOT` — project root (defaults to `../../codex/codex-rs`)
- `RA_SINGLE_CRATE` — e.g. `core/Cargo.toml` to restrict indexing scope

## Output format

JSONL — one JSON object per line, streamed as each file completes:

```json
{"file": "core/src/tools/context.rs", "root": {"name": "core/src/tools/context.rs", "kind": "File", "start_line": 1, "end_line": 283, "line_count": 283, "children": [...]}}
```

Node fields: `name`, `kind`, `detail` (nullable), `start_line`, `end_line`, `line_count`, `children`, and `signature` (nullable, only on Function/Method kinds).

1. **Single-file extraction works perfectly.** Tested with `RA_SINGLE_CRATE="core/Cargo.toml"` on `core/src/tools/context.rs` — full symbol tree extracted correctly.

2. **Batch extraction works for ~341 files.** The `--all-rs --no-signatures` run successfully extracted 341 files (1.7MB of JSONL) covering everything through the first part of `core/` before rust-analyzer died.

3. **Partial output saved at:** `rust-graph/output/symbols_2026-02-16.jsonl` (341 lines, 1.7MB).

4. **Workspace file discovery** correctly finds 901 `.rs` files across 66 workspace crates (via `cargo metadata --no-deps`), excluding dependencies.

## What failed

### Hover/signature extraction hangs

With signatures enabled (the default), the extractor gets stuck around file 19 (`app-server/src/bespoke_event_handling.rs`). Each Function/Method triggers a synchronous `hover` LSP call, and these appear to hang or be extremely slow on the full 930-crate graph. **Workaround:** `--no-signatures` flag skips all hover calls.

### Full batch run OOMs / times out at ~341 files

Even with `--no-signatures`, rust-analyzer accumulates state as files are opened (each `document_symbols` call first does `textDocument/didOpen`). After ~341 files, `documentSymbol` requests start timing out (120s default). The LRU cache is set to 32, but open documents pile up.

**Root cause theory:** Not a system OOM (swap was added but never used). More likely RA's internal memory pressure from having hundreds of documents open simultaneously, causing it to slow down until requests time out.

### Memory situation

- Machine: 7.8GB RAM, was running with 0 swap
- Added 16GB swap file on SSD (`/swapfile`), but swap was never actually used — RA's issues appear to be internal timeouts rather than OS-level OOM
- `RA_SINGLE_CRATE` doesn't reduce the crate graph as expected — still indexes all 930 crates regardless (the `linkedProjects` init option may not work as documented)

## Fix: `didClose` after each file

The original run accumulated open documents — `open_file()` sent `textDocument/didOpen` but never `textDocument/didClose`. After ~341 files, RA's internal state grew until requests timed out.

**Fix:** Added `close_file()` to `LspClient` and call it in `extract_file_symbols()` after each file. With this change, `--no-signatures` mode completes all 901 files successfully.

## Hover/signature extraction: still broken

Hover (`textDocument/hover`) hangs consistently around file ~19 regardless of RA config. Attempted:

1. **Sysroot disabled (`sysroot: null`)** — hangs at file 19. The Chalk trait solver enters pathological recursive loops without `core`/`std` type definitions (`Sized`, `Drop`, `Deref`, etc.). It doesn't fail fast with `{unknown}`, it spins.
2. **Sysroot re-enabled (`sysroot: "discover"`)** + LRU bumped to 128 + hover sub-features disabled (`documentation`, `memoryLayout`, `actions`) — still hangs at file ~19. Adding ~100 sysroot crates wasn't enough; the 930-crate graph with proc macros disabled still produces unresolvable trait constraints.

**Root cause:** `documentSymbol` is purely syntactic (AST parsing). `hover` requires semantic analysis (type inference, trait resolution via Chalk). On a 930-crate graph with proc macros and build scripts disabled, Chalk's recursive solver hits combinatorial explosion on complex function signatures. There is no per-request timeout in RA itself, and `$/cancelRequest` doesn't actually cancel in-flight computations.

**Decision:** Extract symbol trees without signatures (`--no-signatures`). If type signatures are needed later, use tree-sitter for syntax extraction and RA only for targeted hover calls on specific functions of interest, or consider enabling proc macros + build scripts at the cost of much higher memory usage.

## Next steps / ideas

1. **Signatures as a second pass.** Extract all trees without signatures first (done), then do targeted hover calls for specific files/functions of interest with a more permissive RA config.

2. **Consider tree-sitter as alternative.** For pure syntax extraction (no type info), tree-sitter-rust would be instantaneous with zero indexing overhead. Only use RA when type signatures are actually needed.
