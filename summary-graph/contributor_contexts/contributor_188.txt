  - codex-rs/core/src/client.rs: This file manages the communication layer between Codex and AI model providers' APIs. It handles session-level configuration (like authentication and provider selection) that persists across multiple conversation turns, while accepting turn-specific settings (like model choice and reasoning effort) as explicit parameters to keep those lifetimes clear. The file implements two main abstractions: a session-scoped `ModelClient` that holds stable connection setup, and a turn-scoped `ModelClientSession` that manages per-turn state like WebSocket connections and sticky-routing tokens. It supports both HTTP and WebSocket transports for streaming responses, with automatic fallback from WebSocket to HTTP when needed, and handles features like request compression, auth token refresh, and incremental request appending to avoid resending data.
  - codex-rs/core/src/error.rs: This file defines how Codex handles and communicates errors to users. It establishes a comprehensive error system that categorizes different failure modes—like network issues, rate limits, sandbox execution failures, and authentication problems—then formats them into user-friendly messages appropriate for each situation, such as upgrade prompts for free-tier users who hit usage limits or specific recovery instructions for expired credentials.
  - codex-rs/core/tests/suite/client.rs: This file is a comprehensive test suite that verifies how the Codex client sends requests to AI model APIs and processes responses. It tests critical product features like authentication, conversation history management, user instructions, model configuration options (reasoning effort, verbosity), rate limiting, error handling, and resuming saved conversations—essentially ensuring that user inputs are properly formatted and transmitted to the backend with all necessary context and settings.
  - codex-rs/protocol/src/protocol.rs: # Summary

This file defines the communication protocol between Codex (an AI coding agent) and its clients. It establishes the message format for two-way streaming communication: clients send **submissions** (user requests, approvals, configuration changes) and receive **events** (agent responses, command output, status updates). The file also specifies execution policies for sandboxing shell commands, approving actions, and managing file access permissions—allowing clients to control how much freedom the agent has when running code on the user's system.
  - codex-rs/tui/src/status/rate_limits.rs: This file formats rate-limit and credit information for display in the status interface. It takes raw rate-limit data from the API and converts it into human-readable rows that show usage percentages, reset times, and account credit balances—then marks that data as stale if it's older than 15 minutes so users know when information may be outdated.
  - codex-rs/tui/src/status/tests.rs: This file contains automated tests that verify the status display panel—the visual summary shown to users about their current session, token usage, rate limits, and account credits. It tests various scenarios like when reasoning details should appear, how permissions are displayed, whether credits show as unlimited or a specific balance, and how the display adapts to narrow terminal windows.
  - codex-rs/app-server/src/codex_message_processor.rs: # CodexMessageProcessor

This file handles all incoming client requests for Codex's AI assistant features. It processes JSON-RPC messages that control thread (conversation) lifecycle, user interactions, authentication, file searching, and configuration management.

The processor acts as a central hub that translates client API calls into operations on active threads, manages login sessions, handles file discovery, lists available AI models and collaboration modes, and orchestrates event delivery back to clients. It supports both legacy "conversation" APIs and newer "thread/turn" APIs, manages user authentication (API key and ChatGPT), and coordinates real-time event streaming to connected clients listening to specific threads.
  - codex-rs/app-server/tests/common/auth_fixtures.rs: This file provides testing utilities for creating fake ChatGPT authentication credentials. It allows test engineers to easily generate mock login data (tokens, user IDs, email addresses, etc.) and write them to a test environment, so they can verify that the product correctly handles user authentication without needing real OpenAI accounts.
  - codex-rs/app-server/tests/common/lib.rs: This file provides a collection of testing utilities and helper functions that allow engineers to set up realistic test scenarios for the Codex application—such as creating mock servers, fake user authentication, simulated model responses, and configuration files. It acts as a shared toolkit so tests throughout the codebase can consistently simulate the system's behavior without needing to write the same setup code repeatedly.
  - codex-rs/app-server/tests/common/mcp_process.rs: This file provides test infrastructure for communicating with Codex's application server during automated testing. It creates and manages a separate server process that tests can send requests to and receive responses from, simulating real client-server interactions. The file handles the low-level messaging protocol—sending commands like "create a conversation" or "log in," reading back responses, and buffering messages that arrive out of order—so tests can focus on validating the server's behavior without worrying about communication plumbing.
